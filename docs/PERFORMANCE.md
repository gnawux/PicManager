# PicManager 性能分析

本文档以 **M4 Mac + 16GB 内存 + USB 3.1 外置 NVMe SSD** 为参考硬件，分析当前架构的性能瓶颈、规模上限与优化路径。

---

## 硬件基准

| 操作 | 实测/估算值 |
|------|------------|
| USB 3.1 Gen 2 顺序读 | ~700 MB/s（理论上限 1.25 GB/s） |
| USB 3.1 4K 随机读 | ~80–120 MB/s，延迟 ~0.2–0.4 ms |
| SQLite 索引查询（有索引） | O(log n)，单查询 <5 ms（百万行内） |
| M4 软件 JPEG 解码，12MP | ~30–60 ms（image crate 纯 Rust） |
| M4 JPEG 解码（Apple VideoToolbox） | ~5–10 ms（未使用） |

---

## 瓶颈逐项分析

### 1. 缩略图生成（最大瓶颈，与规模无关）

**代码位置**：`src/web/handlers/photos.rs:100` — `generate_thumb()`

**问题一：无缓存。** 每次 `GET /api/photos/:id/thumb` 都从原始文件重新解码。

**问题二：阻塞 async runtime。** `generate_thumb` 是同步函数，直接在 Tokio async worker 线程中调用（未使用 `spawn_blocking`），会阻塞其他异步任务。

每张缩略图耗时分解：

```
读 5 MB JPEG from USB SSD:      ~7 ms
JPEG 解码（12MP，纯 Rust）：    ~40 ms
resize 到 300 px：               ~5 ms
JPEG 编码输出：                   ~8 ms
──────────────────────────────────────
单张合计：                       ~60 ms
```

浏览器 HTTP/1.1 并发 6 个请求，50 张图需要约 9 轮：

```
50 张缩略图 ≈ ceil(50/6) × 60 ms ≈ 500–700 ms / 每次翻页
```

**关键结论：这个耗时与库中照片总数完全无关。** 1 张照片和 100 万张照片，每次翻页速度相同。加缓存是最高 ROI 的优化。

---

### 2. 照片列表查询（弱相关）

**代码位置**：`src/web/handlers/photos.rs:46–65`

两个问题：

**a. `COUNT(*)` 每次翻页都执行：**

```sql
SELECT COUNT(*) FROM photos   -- 全表扫描，每页都执行
```

| 照片数 | `COUNT(*)` 耗时 |
|--------|----------------|
| 1 万 | <1 ms |
| 10 万 | ~3 ms |
| 50 万 | ~15 ms |
| 100 万 | ~50 ms |

单独看不严重，但叠加在每次翻页上会被用户感知。

**b. `OFFSET` 深分页性能退化：**

```sql
SELECT ... ORDER BY taken_at DESC LIMIT 50 OFFSET 500000
```

SQLite 的 `OFFSET` 实现是跳过前 N 行，O(n) 复杂度。在正常浏览模式下（前几十页）几乎不是问题，但若实现「跳转到某日期附近」等功能则需要改为游标分页。

---

---

### 4. GPS 地理编码（与新导入照片数线性相关）

**代码位置**：`src/album/location.rs`

Nominatim API 限速 1 req/s（缓存命中不计）。导入 1,000 张不同 GPS 位置的照片 = 约 **16 分钟** 仅用于地理编码。

影响导入速度，不影响浏览。但地理编码完成前，地点相册不可用。

---

### 5. SQLite 本身（不是瓶颈）

当前库（`photo` 表 + 索引）在此硬件上：

| 规模 | DB 文件大小 | 带索引查询 |
|------|------------|-----------|
| 10 万张 | ~100 MB | <5 ms |
| 50 万张 | ~500 MB | <10 ms |
| 200 万张 | ~2 GB | <20 ms |

SQLite 在 USB NVMe 上支撑 200 万行查询没有问题，不需要换数据库。

## 综合规模上限

| 功能 | 舒适上限 | 瓶颈 |
|------|---------|------|
| **浏览（分页网格 + 相册）** | 无上限（50 万内查询不感知） | 缩略图无缓存（固定 500–700 ms/页） |
| **dedup 操作** | ~5,000 张 | O(n²) + 每对 DB 查询 |
| **导入+地理编码** | 无上限，但每张 GPS 照片慢 1 s | Nominatim 限速 |
| **SQLite 查询** | 200 万+ | 不是瓶颈 |

**一句话结论：当前架构浏览卡不卡与照片总数关系不大，卡的根源是缩略图无缓存（Step 15a）。加缓存后，SQLite 架构可轻松支撑 50 万张照片的日常浏览。**

## 优化路线图

按优先级和实现难度排序：

### 高优先级（每次浏览都感知）

| Step | 内容 | 预期收益 |
|------|------|---------|
| 15a | 缩略图磁盘缓存 + `spawn_blocking` | 翻页从 500–700 ms → <50 ms（重复访问） |
| 15b | `COUNT(*)` 改为计数器表 | 每页节省 3–50 ms |

### 中优先级（超过 5,000 张开始感知）

| Step | 内容 | 预期收益 |
|------|------|---------|
| 16a | dedup 改为增量扫描（只比较新导入） | 单次 dedup 从小时级 → 秒级（增量） |
| 16b | dedup 分桶（pHash 前缀分桶） | 全库重算从 O(n²) → 接近 O(n log n) |

### 低优先级（超过 50 万张或有特定需求）

| 内容 | 触发条件 |
|------|---------|
| 游标分页（`WHERE id < last_id`） | 用户需要按日期跳转、深翻页 |
| 人脸分析改为后台低优先级队列 | 导入时不想等待检测 |
| 地理编码异步化（导入后单独运行） | 导入数千张 GPS 照片时 |

## 参考：未来架构可考虑的方向

这些是假设未来规模远超家庭用途（数百万张）时才需要考虑的，目前无需实施：

- 缩略图用 SQLite BLOB 存储（避免小文件 inode 碎片）
- pHash 相似搜索用 LSH（局部敏感哈希）替代 O(n²)
- 用 `taken_at` 游标分页替代 `OFFSET`
- 多进程读（SQLite WAL 模式已支持）
