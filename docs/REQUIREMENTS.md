我们要做一个功能完善、易于管理、确保照片和相关数据安全的家庭照片与图片管理工具。

## 核心功能

* 支持自动整理照片，提取时间、地点信息、拍摄相机信息；
* 支持重复图片识别，包括图片尺寸缩放后的重复照片，去重功能需要人工确认后再删除；
  * 采用两层去重算法：第一层 Gradient pHash 快速粗筛（O(n log n)），第二层 DCT pHash 精细验证，消除截图 vs 照片等假阳性；
  * 拍摄时间 ≤ 60 秒的照片对（连拍）使用宽松阈值（Hamming ≤ 10），其余使用严格阈值（≤ 3），降低不同日期照片的误报；
  * **CLI（`picmanager dedup`）只负责扫描并报告重复组数量，不在终端提示选择**；确认保留/删除操作须在 Web 界面完成（无法在命令行看图，终端操作不可行）；
* 按照时间和地点、拍摄相机划分相册，可以手工合并；
* 相册侧边栏按三个分类展示：**设备**（camera）、**时间**（time）、**地点**（location）；
  * 每个分类内的相册按该相册最新照片的拍照时间降序排列（无照片的排最后）；
  * 每个分类默认显示 5 个相册，点"更多 (N)"展开其余；
  * 每个分类标题前有折叠三角：展开时 ▼，折叠时 ▶（指向标题名）；点击标题或三角切换折叠状态；

## 格式支持

* 支持多种格式，包括 JPG，索尼的 RAW 格式，苹果的动态照片，网上常见的 webp, png, gif 等格式

## 输入输出

* 从指定目录导入照片文件
* 其他导入方式后续考虑

## 导入行为

* **缺省行为**：将照片 **移动**（move）到 library 中，按拍摄日期放入 `yyyy-mm-dd` 子目录，导入完成后源文件不再保留
* **保留原文件**：提供 `--copy` 参数，改为复制而不删除源文件；适合导入重要原始存档时使用
* **拍摄日期推断优先级**（从高到低）：
  1. **EXIF 字段**（依次尝试，取第一个有效值）：
     - `DateTimeOriginal`（0x9003）— 相机按快门时写入，最可靠
     - `DateTimeDigitized`（0x9004）— 数字化时间，数码相机通常与 Original 相同
     - `GPS DateStamp + TimeStamp` — GPS 授时精度高，但为 UTC，存在时区偏差
     - `DateTime`（0x0132）— 文件最后修改时间，可能被编辑软件改写，最后兜底
  2. **文件名推断**：EXIF 四个字段均无有效值时，从文件名中解析日期时间模式（如 `IMG_20240615_103000.jpg`、`2024-06-15_photo.jpg`），或整体为 Unix 时间戳（秒级或毫秒级）
  3. **无法判断**：照片放入 `unknown/` 子目录，后续通过编辑拍摄时间将其重新归入正确日期目录
* 重复导入同一文件（相同 SHA-256）时跳过，不重复移动或复制

### 大批量分批迁移

适用场景：从 Lightroom 等第三方工具管理的大型照片库迁移时，文件数量可能达数千张，单次导入耗时数小时；需要支持中断后恢复，且源文件不能移动（须配合 `--copy` 使用）。

* **扫描完整列表**：导入开始前先递归扫描源目录，得到全部候选文件列表
* **分批执行**（`--batch-size N`）：每次最多处理 N 张照片；未处理完的文件留待下次运行
* **NDJSON 日志**（`--log <file>`）：每张照片处理完成（成功/跳过/失败）后立即追加一行 JSON 记录；中断后重跑时自动读取日志，跳过已处理的文件；`Failed` 状态不标记为已完成，下次运行自动重试
* **预览模式**（`--dry-run`）：只扫描计数，不实际导入，用于估算工作量
* **幂等保证**：即使日志丢失，数据库 SHA-256 去重仍可防止重复导入，日志仅用于加速跳过

推荐用法（Lightroom 迁移）：

```
picmanager import --copy --batch-size 500 --log import.log /path/to/lightroom/photos
# 中断后重跑同一命令即可恢复，已导入的文件自动跳过
```

## 人脸检测与特征提取

* **自动检测**：每张照片导入时自动进行人脸检测，记录每张人脸的边界框（bounding box）和置信度分数；格式不支持或检测失败时只记录警告，不中断导入流程
* **特征提取**：对检测到的每张人脸提取特征向量（embedding），用于后续的人物识别和相似度计算；特征向量持久化存储，为未来按人物分组相册打下数据基础
* **方向感知检测**：检测前先将图像变换到显示空间（EXIF Orientation + 用户手动旋转/翻转），确保 ArcFace embedding 对人脸朝向不敏感；人脸 bbox 坐标以显示空间存储；升级本功能后需运行 `picmanager faces analyze` 重新分析全库
* **批量重分析**：除导入时自动触发外，支持通过 CLI 和 Web API 手动触发全库或指定照片范围的人脸检测与特征重提取，适用于模型升级或补充分析历史照片的场景
* **本地处理**：所有人脸处理完全在本地运行，不调用任何云端 API，保护家庭照片隐私
* **模型管理**：特征提取依赖预训练 ONNX 模型文件，模型文件独立于二进制分发，首次使用前需下载；提供专用命令拉取模型文件

## WebUI 功能

WebUI 包含四类主要视图：**照片浏览**、**照片详情**、**人物/动物**、**地理位置**，可在视图间导航切换。

### 照片浏览视图

* 以缩略图网格展示照片（300×300 正方形中心裁剪），支持按时间范围、相册筛选
* 点击缩略图进入照片详情视图
* 支持批量框选或逐一勾选照片，对选中照片批量操作，包括：
  * 时间 / 时区调整（写入数据库，不修改原始文件 EXIF）
  * 旋转 / 翻转（写入数据库，不修改原始文件，缩略图自动重新生成）
  * 加入相册等操作（后续扩展）

### 照片详情视图

* 展示大图及完整元信息（EXIF 字段、GPS 坐标、拍摄相机、已检测人脸等）
* 支持编辑部分字段：拍摄时间、时区（仅写数据库，不回写文件）
* **旋转与翻转**：详情视图提供 4 个操作按钮（↺ 左转 90°、↻ 右转 90°、↕ 上下翻转、↔ 左右翻转）；操作结果写入数据库（不修改原始文件），缩略图即时刷新；照片网格选择模式下的批量操作栏提供相同的 4 个旋转/翻转按钮，对所有选中照片批量执行；旋转/翻转操作会使已存储的人脸 bbox 和 embedding 失效（它们是在旧显示空间下计算的），系统在写入 DB 后自动在后台触发受影响照片的人脸重分析，无需用户手动干预；对此修复前已完成旋转的历史照片，可运行 `picmanager faces analyze --rotated-only` 补充修复
* 支持查看原图：点击"查看原图"按钮加载原始文件（可能较大），再次点击切换回缩略图；切换到下一张或关闭模态框时自动复位
* 展示照片中识别到的人物缩略图（人脸裁剪图 + 姓名）

### 人物 / 动物视图

**人物部分**

* 展示经自动聚类后的人物列表，只显示顶级节点（无父节点）；有名字的人物排在无名字人物之前；有子节点的人物卡片显示子节点数量（如"42 张 · 3 个子人物"）；无代表性人脸时显示默认占位图
* 每个人物以代表性人脸缩略图显示（160×160 正方形中心裁剪）
* 自动聚类基于导入时提取的 512 维 ArcFace embedding，使用 DBSCAN 算法（cosine 距离）
* 点击人物缩略图进入该人物的照片相册与详情页
* 支持将多个聚类合并为一个人物（例如同一人的童年与成年期，特征差异较大但实为同一人）
* **合并安全**：将父节点合并到其子节点时，不直接执行会导致子节点 parent_id 自引用（孤儿节点）的操作，而是自动将子节点升级到父节点的位置（继承原父节点的父节点和其余子节点），再完成合并；所有合并操作在前端均需二次确认（含源/目标名称，提示不可撤销），防止误操作
* 人物下支持树状子相册结构（层级无硬性上限），子相册可从人物中移出、重新归属

**人物聚类操作**

* **整合新面孔**（主按钮）：导入结束后自动执行一次；也可手动点击。将所有尚未分配的人脸 embedding 与已有人物比对，距离 < 0.4 则归入最近人物；剩余不匹配的人脸再运行 DBSCAN 创建新人物。**不删除任何现有人物记录**，可安全随时调用
* **全量重建**（次要按钮，需确认）：清空全部 `people` / `person_faces` 记录后重新 DBSCAN 聚类。会丢失所有已命名、合并操作，点击后需二次确认

**人物编辑操作**

* **内联改名**：直接点击人物姓名进入编辑状态，失焦或回车保存；保存前检查重复名称（见下方）
* **单人物扩展菜单**（"…" 按钮）：
  * 忽略此人（`ignored`）：将该人物从常规视图中隐藏，保留人脸数据，可在"已忽略"筛选中恢复
  * 标记为非人物（`not_a_person`）：标记该聚类为误识别，从人物列表彻底隐藏（保留数据供将来模型改善后重分析）
* **多选与批量操作**：支持逐一勾选多个人物聚类；选中的卡片始终显示紫色勾选指示符（不依赖鼠标悬停）；选中后浮出操作栏，支持：
  * 命名并合并：为合并后的人物输入姓名，选择树状结构中的父节点（可留空表示顶级）；合并结果纳入人物树
  * 批量忽略 / 批量标记为非人物
* **操作撤销**：所有人物编辑（改名、合并、忽略、标记为非人物、树状结构调整）均可点击撤销；前端维护操作历史栈，撤销时调用逆操作 API，防止误操作

**人物详情页**

* 点击人物缩略图进入详情页：展示人物姓名、代表性头像、所属照片网格（分页）
* **树状结构编辑**：可在详情页设置或更改该人物的父节点（搜索并选择已有人物），或将该人物提升为顶级
* **移出操作**：可将该人物或其某棵子树从当前父节点中移出，重新归为顶级或另一人物的子节点；移出仅改变树结构，不删除任何人脸或照片数据
* **「合并到…」对话框**：工具栏「合并到…」按钮弹出候选列表，同时拉取 merge-suggestions 得到相似度数据；候选人物按**相似度降序**排列（有相似度数据的排在前面，其余按照片数排列），每个有相似度数据的候选项右侧显示"N% 相似"

**照片选择模式**

* 详情页工具栏提供「选择照片」切换按钮；激活后，照片网格中每张照片显示勾选框（与照片浏览视图样式一致）
* 选中照片后底部浮出操作栏，显示：已选 N 张 ＋「创建子人物」＋「转移至兄弟」（仅当当前人物有父节点时可用）＋「取消」
* 退出详情页时自动清空选择状态

**照片选择操作一：创建子人物**

* 选中一组照片 → 点击「创建子人物」→ 弹出命名对话框（触发重名检测）
* **重名"是同一人"**：将已有同名人物设为当前人物的子节点，并将选中照片的人脸从当前人物转移到该已有人物
* **重名"不同人"** / 无重名：新建人物（parent_id = 当前人物 id），将选中照片对应的人脸从当前人物转移到新人物
* 若选中照片在当前人物下无对应人脸记录，人脸数为 0；仍创建子人物节点（允许空容器节点），跳过转移
* 操作完成后刷新照片网格和子人物面板
* 支持撤销：将人脸转移回当前人物 + 删除新建的空人物节点

**照片选择操作二：创建父节点**

* 面包屑栏提供「创建父节点」按钮（与「更改父节点」并列）
* 点击 → 弹出命名对话框（触发重名检测）
* **重名"是同一人"**：仅将当前人物的 parent_id 改为已有同名人物（该已有人物成为父节点），不新建人物
* **重名"不同人"** / 无重名：新建人物（parent_id = 当前人物的原 parent_id），将当前人物的 parent_id 改为新人物；新父节点仅作树结构容器，无人脸
* 操作完成后刷新面包屑
* 支持撤销：恢复当前人物原 parent_id + 删除新父节点（若为新建节点）

**照片选择操作三：转移至兄弟节点**

* 仅当当前人物有父节点时可用
* 选中照片 → 点击「转移至兄弟」→ 弹出兄弟选择器，列出同父节点下的其他人物及「新建兄弟人物」选项
* **选择已有兄弟**：将选中照片的人脸从当前人物转移到目标兄弟
* **新建兄弟**：弹出命名对话框（触发重名检测）；重名"是同一人"则转移到该已有人物；"不同人"/无重名则新建（parent_id = 当前人物的 parent_id）再转移
* 转移后刷新当前人物照片网格
* 支持撤销：将人脸转移回当前人物 + 删除新建的空兄弟节点

**重复姓名检测**

* 命名或改名时，若输入的姓名与已有人物完全相同，弹出确认对话框
* 对话框并排展示当前人物与已有同名人物的代表性人脸缩略图
* 用户可选择：
  * "是同一人" → 根据操作上下文执行对应动作（改名/聚类合并时合并人脸；创建子/父/兄弟节点时复用已有人物）
  * "不同人" → 保留重名，各自独立建档

**动物部分**

* 使用通用目标检测模型（如 YOLO 类）在导入时检测照片中的动物，记录动物种类（猫 / 狗 / 鸟等）
* 动物视图按种类分组展示含有动物的照片
* 不做动物个体识别（即不区分"是否为同一只宠物"），原因：个体识别所需模型与聚类方案的实现代价显著高于种类检测

### 地理位置视图

* 数据来源：Nominatim 反地理编码返回的行政区划层级；geocache 扩展存储完整层级（国家 → 州/省 → 城市 → 区县）
* **中国直辖市处理**：北京、上海、天津、重庆作为省级行政区放在"州/省"层级；区级名称（如西城区）放在"城市"层级。Nominatim 对直辖市不返回 state 字段，通过 ISO 3166-2 代码（如 CN-BJ）推断
* 已存在的地理编码记录若 state 为空（旧版数据）、或 city/state/country 三字段均为 NULL（历史上 Nominatim 临时失败写入的空记录），均可通过 Web 界面"再次地理编码"自动修复
* **邻近坐标复用**：geocache 作为自增长本地库，存储历次地理编码结果。导入时对每个未命中精确 key 的 GPS 坐标，先在 ±0.01°（约 1 km）范围内查找已有的有效缓存条目；命中则直接复用并写入精确 key，无需调用 Nominatim。仅当邻近范围内也无有效缓存时才发起 API 请求。实测此策略可覆盖约 89% 的坐标，大幅减少 API 调用次数
* **层级列表**：按行政区划层级钻取（如 USA → California → San Francisco）；点击任意层级（国家、省/州、城市）均显示该层级聚合的所有照片；点击国家显示该国全部照片，点击省/州显示该省全部照片，点击城市显示该城市照片
* **Unknown 条目**：geocache 中 city/state 字段为 NULL 的照片在对应层级显示为 "Unknown"，点击后可查看这些无精确地名的照片（如仅知国家/省但无城市信息）
* **地图打点**：在地图上标注照片 GPS 位置，支持缩放和聚合显示；地图为基础功能，层级列表优先实现
* "湾区"等非正式地理概念不做特殊处理，通过行政区（County 级）自然覆盖；用户可通过手动创建相册来组织跨行政区的地理分组

### 元数据补全

侧边栏提供两个独立的补全按钮，可分别触发，互不干扰：

* **补全人脸识别**：对从未被分析过（`faces` 表中无记录）的照片触发人脸检测与特征提取；利用现有异步 face job 系统，实时展示已处理/总数进度；完成后若当前为人物视图则自动刷新
* **补全地理编码**：对有 GPS 坐标但尚未完成 Nominatim 反地理编码（`geocache` 表中无对应条目）的照片触发地理编码；地理编码在后台异步执行，前端轮询运行状态；完成后若当前为地点视图则自动刷新

## 命令行导入进度输出

导入命令 `picmanager import <dir>` 支持大批量照片场景下的实时进度输出，分三段显示导入、人脸分析、地理编码的独立进度：

* **进度打印**：每分钟打印一次，格式：
  `[00:01:00] 导入：123/1000 (12%) ｜ 人脸：45 张 ｜ 地理：等待中`
  文件全部导入完成后，地理编码阶段开始：
  `[00:02:00] 导入：1000/1000 (100%) ｜ 人脸：45 张 ｜ 地理：3/7 (43%)`
  若新导入照片均无 GPS 则显示 `地理：无 GPS`
* **汇总信息**：全部任务完成后打印带耗时的汇总：
  `完成（耗时 N 分 M 秒）：共 X 张，导入 Y，跳过 Z，失败 W`
* **范围限定**：地理编码只处理本次新导入的照片（非全库），避免对库内存量 GPS 照片的重复编码拖慢导入
* **行为一致性**：与 `fill-missing` 保持相同的轮询节奏（每 5 秒检查一次、每 60 秒打印一次），完成时立即打印

## 命令行元数据补全

除 Web UI 外，提供命令行方式一次性对全库进行元数据补全，适合首次下载模型后或脚本化运行：

```
picmanager fill-missing [--faces] [--geo]
```

* **默认行为**（不带任何标志）：同时补充人脸和地理两类缺失元数据
* `--faces`：仅补充缺少人脸分析记录的照片
* `--geo`：仅对有 GPS 但尚无反地理编码缓存的照片触发地理编码
* **进度输出**：每分钟打印一次当前进度，格式：`[00:01:00] 人脸：12/75 (16%) ｜ 地理：3/23 (13%)`；任务完成时也打印最终状态
* **汇总信息**：全部任务完成后打印汇总，包含：分析的照片数、跳过（已有记录）数、新增人脸数；编码成功数、失败数（无城市信息）、总待处理数

## 人物聚类辅助功能

### 人物列表排序

* 已命名人物排在未命名人物之前（保持现有行为）
* **两组内部均按照片数降序排列**：照片多的人物优先显示，便于优先处理高频人物

### 聚类合并建议

* 针对**已命名的人物**，在详情面板展示「建议合并」区（默认折叠，点击标题展开）
* 系统计算该命名人物与所有其他 active 人物的人脸 embedding 质心相似度，按相似度排序给出最多 **20** 条建议
* 质心使用置信度优先的精炼算法计算（见下文"质心算法"）
* 每条建议显示：80×80 人脸缩略图（紫色边框，点击可放大）、姓名或"未命名"、照片数、相似度百分比
* 建议列表按**已命名 / 未命名**分组展示：已命名人物排在前方，再跟一个分隔符，未命名人物排在后方；两组内部均按相似度降序排列；条目超出一行宽度时横向滚动
* 支持直接从建议列表点击「合并」发起合并操作，合并后刷新照片列表和人物列表

### 疑似错误聚类检测与质心诊断

* 在人物详情面板展示「⚠ 可能误入的照片」区（默认折叠，点击标题展开）；只要该人物有人脸 embedding 即显示此区域
* 区域内包含：
  * **质心统计行**：显示质心使用的人脸数 / 总 embedding 数，以及全部人脸到质心的相似度分布（最近·P25·中位·P75·最远），颜色编码（≥80% 绿、≥65% 黄、≥50% 橙、<50% 红），用于快速判断聚类紧密程度
  * **「离群脸诊断」按钮**：打开诊断模态窗，展示最多 40 张按相似度从低到高排列的人脸缩略图（不设距离阈值），支持：
    * 点击缩略图选中/取消（Shift+点击放大）；「全选」/「取消全选」批量操作
    * 「创建子人物」：将选中的人脸所属照片作为子人物的来源，走现有命名+父子关系流程（适用于将有遮挡的照片归入专属子人物）
    * 「单独移出」：将单张人脸移出当前人物，自动为其新建未命名人物记录
  * **离群脸列表**：仅展示距质心相似度 ≤ 50%（余弦距离 > 0.50）的人脸，最多 5 条，80×80 缩略图（红色边框），每条可「移出」或「保留」
* 在照片网格中，参与质心计算的照片缩略图下方显示浅黄底色，便于判断质心代表性

### 质心算法

系统使用两步精炼算法计算人物的代表性质心：

1. **置信度预过滤**：优先选取检测置信度 ≥ 0.85 的人脸（不足 10 张时降至 ≥ 0.70，仍不足则用全部）。此步骤排除暗光、畸变、侧脸等低质量检测，避免其 embedding 拉偏质心
2. **几何精炼**：对于过滤后仍有 50+ 张人脸的人物，先计算粗质心，再取距粗质心最近的 40% 作为精炼候选，最终质心由这 40% 计算。适用于照片条件差异大的人物（不同年龄段、多种角度）

所有需要质心的操作（合并建议、离群脸检测、质心统计）均使用此算法。

## 精选集

精选集是用户手工创建的照片集合，与按时间/地点/设备自动生成的相册并列显示于侧边栏。

* **创建与管理**：用户可随时新建精选集（输入名称），支持改名和删除；删除精选集只删除集合关系，不影响照片本身
* **添加照片**：可在任意视图（照片浏览、相册、人物详情）进入多选模式，选中照片后点"加入精选集"批量加入；也可点"添加整个相册到精选集"将当前相册所有照片一键加入；一张照片可加入多个精选集
* **移除照片**：在精选集视图内多选照片后点"从精选集移除"批量移除；移除只删除关联关系，不影响照片本身
* **浏览**：点击侧边栏精选集条目，主区域展示该精选集中的照片（分页，按拍摄时间排序）

### 前置要求（首次使用）

> **重要**：PhotoKit 导出云端照片时，Photos 框架会先将原始文件下载到 Photos Library 所在磁盘。若 Photos Library 仍在系统卷（默认 `~/Pictures/`），下载大量 iCloud 照片可能占满系统卷。
>
> **推荐做法**：在 macOS Photos.app 偏好设置 → 通用 中，将照片库迁移到与 PicManager 库同一块外置硬盘，再设置为系统照片库。

### 格式处理规则

| 资产类型 | 导出内容 | 跳过内容 |
|----------|----------|----------|
| 普通 JPEG / HEIC | 静态图文件 | — |
| Live Photo | 静态图（HEIC/JPEG） | 视频组件（.mov） |
| RAW＋JPEG 组合 | JPEG 文件 | RAW 文件（.dng / .arw 等） |
| 连拍（Burst） | 精选张（userPick） | 其余连拍帧 |
| 视频 | 跳过 | — |

### 与 PicManager 批量导入集成

PhotoBridge 调用 `picmanager import` 子进程，两侧使用相同的 `--batch-size` 参数：

```bash
# 首次全量导出（Photos Library 迁移完毕后执行一次）
photobridge export --batch-size 200

# 日常增量同步（新照片加入 iCloud 后执行）
photobridge sync --batch-size 200

# 查看上次同步时间与待处理数量
photobridge status
```

PhotoBridge 解析 `picmanager import --log` 输出的 NDJSON 日志，确认每批照片成功入库后再清空暂存文件，失败的文件在下次运行时自动重试。

### HEIC 方向修正

Apple Photos 允许用户在 Photos.app 中手动旋转照片，但不会修改 HEIC 文件本身，只写入私有内部数据库。PhotoBridge 导出的 HEIC 文件 EXIF 方向标签可能与 Photos 的显示方向不一致，导致 PicManager 显示方向错误。

**导出时自动修正（推荐）：**
`export` / `sync` 使用 `writeAssetResourceOrientationFixed`，在写出文件后自动查询 Photos 的显示方向（`PHImageManager.requestImageDataAndOrientation(version:.current)`），若与文件 EXIF 不一致则用 `exiftool` 无损修改 EXIF 标签。需要安装 `exiftool`（`brew install exiftool`），未安装时静默跳过。

**已导出/已导入文件的批量修正：**
`photobridge fix-orientations` 命令批量处理已导出的 HEIC 文件：

```bash
# 检查 staging 目录中方向不一致的文件（不修改）
photobridge fix-orientations --dir /path/to/staging --dry-run

# 修复 staging 中的 HEIC 文件
photobridge fix-orientations --dir /path/to/staging

# 修复已导入 PicManager 的 HEIC，同时更新 DB 和清除缩略图缓存
photobridge fix-orientations --dir /path/to/library \
  --picmanager-db /path/to/picmanager.db \
  --thumbs-dir /path/to/.thumbs
```

通过文件名反向解析 `localIdentifier`，批量查询 Photos 资产（每批 500 个），并发查询方向（`--max-concurrent`，默认 16）。不在本机 Photos 库中的文件（iCloud 仅存储、受限访问）会跳过。

### 状态管理

PhotoBridge 在 `~/Library/Application Support/PhotoBridge/state.json` 维护：

* **增量变更令牌**（`PHPersistentChangeToken`）：记录上次同步位点，`photobridge sync` 从此处开始，只处理新增资产
* **已导出资产 ID 列表**：防止因 picmanager 入库失败导致重复导出

### 技术规格

* 语言：Swift，面向 macOS 26+
* 框架：PhotoKit（系统照片库访问）
* 权限：`NSPhotoLibraryUsageDescription`（完整照片库读取权限）
* 首次运行时弹出系统授权对话框，后续无需重复授权

## 运动记录（2.0）

### 概述

PicManager 2.0 将照片管理扩展为生活记录工具，首先支持户外运动数据的导入与浏览。用户可批量导入 Garmin 等设备的运动文件，在专属的「运动」标签页中浏览运动列表与轨迹详情，系统自动将运动时段内拍摄的照片标注在运动地图上，形成图文结合的运动日记。

### 支持的运动格式

| 格式 | 说明 | 扩展名 |
|------|------|--------|
| **FIT**（Flexible and Interoperable Data Transfer） | Garmin 专有二进制格式，包含完整运动数据（GPS轨迹、心率、踏频、功率、高度等） | `.fit` |
| **GPX**（GPS Exchange Format） | 开放 XML 格式，广泛兼容各品牌 GPS 设备与 App（Garmin Connect / Strava 导出） | `.gpx` |

支持的运动类型（从文件元数据读取，无法识别时归为 `other`）：跑步（running）、徒步（hiking）、骑行（cycling）、步行（walking）、越野跑（trail_running）、游泳（swimming）、其他（other）。

### 运动文件目录

PicManager 维护一个专用的运动文件目录（在配置文件中指定，默认 `{library}/.activities/`），用于集中存放所有导入的 FIT/GPX 原始文件。导入时将源文件**复制**到此目录，数据库 `source_path` 记录目录内路径（而非用户的原始下载路径）。目录结构按年份组织：`{activities_dir}/{yyyy}/{filename}`。

### 导入

**CLI 命令：**

```bash
picmanager activities import <file_or_dir>            # 导入单文件或目录下全部 FIT/GPX
picmanager activities import --dry-run <file_or_dir>  # 仅统计数量，不写库
picmanager activities sync-usb                        # 从 USB 连接的 Garmin 设备导入新文件
picmanager activities sync-usb --device GARMIN        # 指定 USB 卷名（默认自动检测）
picmanager activities update-titles [--dry-run]  # 为 title 为空的运动补全自动标题
picmanager activities fix-metadata [--dry-run]   # 从已保存的 FIT 文件重新读取设备名称和传感器信息
```

行为规范：

* 按文件 **SHA-256** 去重，重复导入同一文件静默跳过
* 递归扫描目录，按扩展名（`.fit` / `.gpx`，大小写不敏感）识别格式
* 单文件解析失败只记录 `warn`，不中断批量导入，最终汇总失败数量
* 源文件**复制**到 `activities_dir`，不删除原始文件
* `sync-usb` 自动检测 `/Volumes/` 下挂载的 Garmin 设备（卷名含 `GARMIN`），读取 `{volume}/GARMIN/Activity/` 目录
* 导入完成后打印汇总：`共 N 个文件，导入 X，跳过（重复）Y，失败 Z`

> Garmin 数据同步方案详见 `docs/GARMIN_SYNC.md`。MVP 阶段使用手动导入 + USB 直连，非官方 API 自动同步作为后续可选增强。

### 数据模型

**`activities` 表（每次运动一条记录）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | |
| `sha256` | TEXT UNIQUE | 文件内容哈希，用于去重 |
| `source_path` | TEXT | 原始文件路径（仅记录，不强依赖） |
| `file_format` | TEXT | `'fit'` / `'gpx'` |
| `title` | TEXT | 活动名称（GPX `<name>` 字段），FIT 文件无标题字段；`NULL` 时按 `{运动类型}-{MM-DD}-{距离}@{城市}` 自动生成 |
| `activity_type` | TEXT | `running` / `hiking` / `cycling` / `walking` / `trail_running` / `swimming` / `other` |
| `start_time` | TEXT | UTC ISO-8601，轨迹第一个点的时间 |
| `end_time` | TEXT | UTC ISO-8601，轨迹最后一个点的时间 |
| `duration_seconds` | INTEGER | 总运动时长（不含暂停） |
| `distance_meters` | REAL | 总距离（米） |
| `elevation_gain_meters` | REAL NULL | 累计爬升（米），无高度数据时为 NULL |
| `avg_heart_rate` | INTEGER NULL | 平均心率（bpm），无数据时为 NULL |
| `max_heart_rate` | INTEGER NULL | 最大心率（bpm） |
| `calories` | INTEGER NULL | 消耗卡路里 |
| `device` | TEXT NULL | 设备名称（如 `"Garmin Fenix 7S"`） |
| `import_status` | TEXT | `'imported'` / `'failed'` |

**`activity_track_points` 表（每个 GPS 轨迹点一条记录）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | |
| `activity_id` | INTEGER FK | → activities(id) ON DELETE CASCADE |
| `ts` | TEXT | UTC ISO-8601 时间戳 |
| `lat` | REAL | WGS84 纬度 |
| `lon` | REAL | WGS84 经度 |
| `elevation` | REAL NULL | 海拔（米） |
| `heart_rate` | INTEGER NULL | 心率（bpm） |
| `cadence` | INTEGER NULL | 踏频 / 步频（rpm/spm） |
| `speed` | REAL NULL | 速度（m/s） |

轨迹点数量估算：典型 1 小时跑步约 3600 点（1 秒采样），一年 200 次运动约 72 万点，存储可控。

**照片关联（不设独立表，按需查询）：**

照片与运动的关联采用**双重过滤**，两个条件均须满足：

1. **时间**：照片 `taken_at` 落在 `[activity.start_time, activity.end_time]` 区间内
2. **GPS 距离**：照片须有 GPS 坐标，且距运动轨迹最近点 ≤ 500 米

纯时间过滤会把同一时段不同位置的家庭成员照片误关联进来（如一人跑步时另一人在家拍照），GPS 距离过滤是必要条件。无 GPS 的照片不参与关联。

关联在展示时即时计算，无需预存：

```sql
-- 伪码：先按时间筛，再在应用层按距轨迹最近点过滤
SELECT * FROM photos
WHERE taken_at BETWEEN :start_time AND :end_time
  AND gps_lat IS NOT NULL AND gps_lon IS NOT NULL
ORDER BY taken_at
-- 然后在应用层计算每张照片距轨迹的最小距离，过滤掉 > 500m 的
```

**左侧：轨迹地图**

* 使用 Leaflet 展示 OpenStreetMap 底图
* 将轨迹点连线绘制为彩色折线（颜色可按心率或速度渐变，初版用固定颜色即可）
* 地图自动缩放适配轨迹包围盒（`fitBounds`）
* 在轨迹上标注关联照片（使用小相机图标 📷），点击显示该照片缩略图弹窗；弹窗点击可跳转到照片详情
* 无 GPS 的关联照片不在地图上显示（仅出现在右侧列表）
* 地图支持缩放、拖拽

**右侧：运动元数据 + 关联照片**

元数据区展示：

| 项目 | 说明 |
|------|------|
| 运动类型 | 图标 + 文字 |
| 开始时间 | 本地时区，精确到分钟 |
| 时长 | h:mm:ss |
| 距离 | X.XX km |
| 配速 / 速度 | 跑步/徒步显示配速（min/km），骑行显示速度（km/h） |
| 累计爬升 | X m（无数据时隐藏） |
| 平均/最大心率 | X / X bpm（无数据时隐藏） |
| 卡路里 | X kcal（无数据时隐藏） |
| 设备 | 设备名称（如 `Garmin Fenix 7S (v12.20)`，无数据时隐藏） |
| 传感器 | 外接传感器列表（类型图标 + 名称 + 电量，无传感器时隐藏） |
| 原始文件 | 文件名（格式标注） |

关联照片区：

* 标题「本次运动中的照片（N 张）」
* 以缩略图网格展示（同照片浏览视图样式）
* 按拍摄时间排序
* 点击照片跳转到照片详情视图
* 无关联照片时显示「本次运动无关联照片」

#### 运动剪辑（Trim）

运动详情的元数据区底部提供「✂ 剪辑运动」入口，用于裁剪忘停表等导致的开头/结尾冗余数据。

**交互流程：**

1. 点击「✂ 剪辑运动」打开全屏剪辑弹窗
2. 顶部切换坐标轴（**时间** / **距离**）
3. 三个数字实时展示：总时间 / 修正开始（截掉的开头长度）/ 修正结束（截掉的结尾长度）
4. 中间 Leaflet 地图：蓝色折线 = 保留段，灰色折线 = 截掉段，随把手拖动实时更新
5. 底部图表（海拔或配速折线图）：拖动左把手调整起点，拖动右把手调整终点
6. 点击「保存剪辑」弹出**不可撤销确认框**（显示截掉的开头/结尾时长），确认后写库

**保存行为：**

* 删除选定范围外的所有轨迹点（`activity_track_points`）
* 重新计算并更新 `activities` 表的 `start_time`、`end_time`、`duration_seconds`、`distance_meters`、`elevation_gain_meters`、`avg_heart_rate`、`max_heart_rate`
* 原始文件（FIT/GPX）保留不变，数据库为剪辑后的唯一数据源
* 操作不可撤销；保存后自动刷新运动详情页

#### 运动合并（Merge）

运动列表页支持多选并合并，适用于同一运动分两段记录（如分段计时、停表后重新开始）的场景。

**交互流程：**

1. 点击列表右上角「多选」按钮进入多选模式
2. 点击运动行切换选中状态，底部工具栏实时显示「已选 N 个运动」
3. 选中 ≥2 个**同类型**运动后「合并运动」按钮可用（不同类型时置灰并提示）
4. 弹出合并确认框：显示各段列表、合并后统计预览（时长含间隔、总距离、总爬升）、可编辑标题、不可撤销警告
5. 点击「确认合并」，成功后自动退出多选并刷新列表

**校验规则：**

* 至少选择 2 个运动（否则合并按钮不可用）
* 所有选中运动必须为**同一类型**（跑步/骑行等）
* 任意两段时间不可重叠（有重叠时 API 返回 400 time_overlap）
* 所有选中运动必须有明确的 start_time 和 end_time

**合并后统计：**

| 字段 | 计算方式 |
|------|---------|
| `start_time` / `end_time` | 所有段的最早/最晚时间 |
| `duration_seconds` | 各段时长之和（不含间隔空档） |
| `distance_meters` | 各段距离之和 |
| `elevation_gain_meters` | 各段爬升之和 |
| `avg_heart_rate` | 各段按时长加权平均 |
| `max_heart_rate` | 取最大值 |
| `calories` | 各段之和 |
| `file_format` | `"merged"` |

**保存行为：**

* 新建一条 `import_status='imported'` 的合并记录
* 原始轨迹点全部迁移到新记录（`activity_id` 更新）
* 源记录软删除：`import_status='merged'`（sha256 保留，防止重复导入同一文件）
* 操作不可撤销

### 关联照片的触发时机

| 事件 | 操作 |
|------|------|
| 导入运动文件时 | 无需额外操作，关联是即时查询，不缓存 |
| 导入新照片后 | 同上，下次打开运动详情时自动包含新照片 |
| 修改照片拍摄时间后 | 自动生效（查询使用最新 taken_at） |

### 边界情况与约束

* **时区**：FIT/GPX 轨迹时间为 UTC，照片 `taken_at` 也按 UTC 存储（含时区偏移字段）；关联查询统一用 UTC 比较，不受设备时区设置影响
* **时钟偏差**：相机与 GPS 设备时钟可能存在数分钟偏差；初版不做额外补偿（用户可通过「编辑拍摄时间」手动修正照片时间）
* **无 GPS 的运动文件**（纯心率记录等）：可以导入，元数据正常存储，地图区显示"此运动无 GPS 轨迹数据"，关联照片仍按时间段展示
* **重叠时段**：多次运动时段重叠时，一张照片可同时关联多个运动（各自独立展示，不做去重）
* **轨迹点采样过密**：前端绘制时，若轨迹点超过 **7200 个**（约 2 小时 × 1 点/秒），使用 Ramer-Douglas-Peucker 算法降采样后再传给 Leaflet（避免前端渲染卡顿）；服务端提供采样后的轨迹数据接口；原始轨迹点完整保存在数据库，降采样仅用于展示

### API 设计（概要）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/activities` | 运动列表（分页，支持 type 过滤） |
| GET | `/api/activities/:id` | 运动详情（含元数据） |
| GET | `/api/activities/:id/track` | 轨迹点数组（自动降采样） |
| GET | `/api/activities/:id/photos` | 关联照片列表 |
| POST | `/api/activities/:id/trim` | 剪辑运动（删除范围外轨迹点并重新计算统计数据） |
| POST | `/api/activities/merge` | 合并多段运动（同类型、无时间重叠，源记录软删除） |
